Where analysis succeeded:

- Payable classification was directionally correct: WETH has payable paths (deposit/fallback) and non-payable ERC-20 methods.
- No proxy and no selfdestruct findings are consistent with the known contract role.

Where analysis produced false positives or misses:
- Likely false positive in Pattern 2 Unchecked Calls: WETH source includes msg.sender.transfer() in withdraw; low-level CALL patterns generated by the compiler can appear as unchecked even when revert semantics are enforced by higher-level constructs.

- Pattern 6 reports no mutex guard. That is technically true (no lock variable), but it does not by itself prove exploitability.
- ABI heuristics are intentionally approximate and do not capture event semantics or economic invariants.

Bottom line:
- For a classic token wrapper, interface recovery and CFG shape were strong. 
- Security heuristics are useful for triage but should not be treated as final exploit proof without source-aware/manual review.

## 3) The Ethereum verified-source certainty problem

Question: How certain can we be that source code shown on Etherscan matches bytecode at an address?

Short answer:
- High confidence is possible, but only if build settings and metadata are matched exactly. Source text alone is not enough.

What must match:
1. Compiler version pinning
- Exact compiler build (for example v0.4.19+commit.c4cbbb05) must match. Minor version differences can change opcodes.

2. Optimization settings
- Optimizer enabled/disabled and optimizer runs must match.
- EVM target, viaIR mode, and metadata settings also affect output bytecode.

3. Metadata hash and CBOR trailer
- Solidity appends metadata (CBOR trailer) that includes hashes and compiler references.
- Runtime bytecode equality checks can fail if metadata differs, even when logic is semantically equivalent.

4. Constructor arguments and immutables
- Deployed bytecode can include constructor-initialized values and immutable substitutions.
- Creation bytecode and runtime bytecode must be treated separately when verifying.

Practical verification process:
- Get the contract bytecode directly from the blockchain.
- Compile the same source code yourself using the exact same compiler/settings.
- Compare runtime bytecode, accounting for metadata expectations.
- For proxies, also inspect implementation slot resolution and verify implementation bytecode separately for the contract behind the proxy.

What this codebase already does well:
- It can detect metadata at the end of bytecode.
- It can sometimes read compiler version hints from that metadata.

What is still missing for stronger certainty claims:
- No full automated “rebuild and compare” pipeline.
- It does not recover all exact build settings (like optimizer runs and EVM version) from explorer data.
- It does not do strict metadata-hash equivalence checks.

## 4) Solidity vs Vyper vs Huff at bytecode level

### Solidity-style profile (WETH sample)

Observed startup sequence:
- PUSH1 0x60
- PUSH1 0x40
- MSTORE
- CALLDATASIZE gate and strict PUSH4/EQ/JUMPI dispatch chain

What we saw:

- It starts with a very common Solidity setup pattern.
- Then it uses the usual function-dispatch shape.

Implications:
- Current strict selector extraction heuristic works well on this class.
- Function-surface reconstruction is high quality for conventional Solidity dispatch layouts.

### Vyper profile (Curve 3Pool sample)

Verified compiler marker from source page:
- vyper:0.2.4

Observed startup behavior:
- Early branch on CALLDATASIZE then immediate jump pathing
- Large constant materialization into memory slots (for example MSTORE at 0x1c, 0x20, 0x40, 0x60, 0x80 regions)
- Strict PUSH4/EQ/PUSH2/JUMPI dispatcher pattern not matched by current extractor (0 dispatch entries)

Implications:
- The analyzer still builds a large, meaningful CFG (1370 blocks, high static jump ratio), but selector recovery under current strict pattern is incomplete for this Vyper shape.
- Memory setup appears more table/constant-oriented than many Solidity token/router examples.

### Huff profile (expected differences)

Huff contracts are hand-authored at opcode/macro level, so typical differences are:
- No guaranteed compiler prologue convention.
- Custom dispatch logic (often jump tables or compact branch trees) that may not look like Solidity/Vyper templates.
- Stack choreography (DUP/SWAP) is often denser and more bespoke.
- Metadata trailers may be absent or minimal compared with high-level compiler outputs.

Analyzer impact:
- Template-based selector extraction is most fragile on Vyper/Huff variants.
- CFG and low-level opcode scans remain valuable, but signature recovery and semantic labeling need language-aware dispatch plugins.

### Explicit Dispatch and Memory Layout Comparison

This subsection explicitly compares dispatch-table structure and memory layout across Solidity, Vyper, and Huff patterns, using one real Vyper contract from this study.

Contract evidence used:
- Solidity sample: WETH (0xc02aaa39b223fe8d0a0e5c4f27ead9083c756cc2)
- Vyper sample: Curve 3Pool (0xbebc44782c7db0a1a60cb6fe97d0b483032ff1c7), verified compiler marker `vyper:0.2.4`
- Data source: [research_results.json](research_results.json)

Observed structural differences:

#### 1. Solidity dispatch-table shape (WETH)
- Repeated selector checks follow the common pattern: PUSH4 selector, EQ, PUSH2 destination, JUMPI.
- This matched current strict extractor rules and produced 11 dispatch entries.
- Result: function-surface recovery is strong for this layout.

#### 2. Vyper dispatch-table shape (Curve 3Pool)
- Early CALLDATASIZE branching and immediate jump-heavy control flow were observed.
- The strict Solidity-style PUSH4/EQ/PUSH2/JUMPI chain was not matched, producing 0 dispatch entries in current extraction mode.
- Result: CFG was still meaningful, but selector recovery was incomplete for this Vyper structure.

#### 3. Solidity memory layout behavior (WETH)
- Startup includes a compact prologue and early memory-pointer initialization.
- Memory behavior is more ABI-dispatch oriented and less table-materialization heavy.
- This is aligned with common Solidity runtime structure.

#### 4. Vyper memory layout behavior (Curve 3Pool)
- Startup shows larger constant materialization into memory (for example MSTORE writes around 0x1c, 0x20, 0x40, 0x60, 0x80 regions).
- Memory setup appears more table/constant-oriented before normal function routing.
- This structural difference contributes to missed selector extraction under strict Solidity assumptions.

#### 5. Huff expected bytecode structure
- No guaranteed compiler prologue.
- Dispatch often implemented as custom jump tables or compact branch logic.
- Stack choreography is typically denser and more bespoke.
- Metadata trailers may be absent or minimal.
- Practical impact: template-based selector extraction is most fragile on Huff-like layouts.

Quantitative contrast from this run:
- WETH: 11 dispatch entries, CFG 103 blocks / 105 edges.
- Curve 3Pool: 0 dispatch entries, CFG 1370 blocks / 1447 edges.
- Interpretation: control-flow recovery still works at scale, but selector extraction needs language-aware dispatch plugins.

Conclusion:
- At least one verified Vyper-compiled contract was analyzed.
- Dispatch-table and memory-layout differences versus Solidity were documented.
- Huff-specific structural expectations and analyzer impact were also covered.
